1. Общая идея архитектуры

В XFS4IoT у нас есть два уровня сервера:

1) Service Publisher
Это “точка входа” для discovery. Клиент сначала подключается сюда и спрашивает:
какие сервисы вообще опубликованы?
2) Service Provider конкретного устройства
Например SimCashAcceptor. После discovery клиент уже подключается к URI конкретного сервиса и работает с ним через Common.Capabilities, Common.Status, Storage.GetStorage, CashAcceptor.* и так далее. 
Это видно и по клиентской форме: сначала Service Discovery, потом клиент получает service URL и уже дальше общается с сервисом устройства.

То есть поток такой:
Client -> Publisher(GetServices) -> serviceURI -> конкретный ServiceProvider(Common.*, Storage.*, CashAcceptor.*)

3. Документация по основным классам
ServicePublisher
Что это

Это discovery-сервис верхнего уровня.

Задача
занять один из стандартных XFS4IoT портов
поднять publisher endpoint, обычно /xfs4iot/v1.0/
отвечать на ServicePublisher.GetServices
хранить список опубликованных сервисов
помочь клиенту найти URI конкретного сервиса
Что делает в твоем коде
выбирает URI и порт
создает EndPoint
регистрирует GetServicesCommand
регистрирует GetServiceHandler
отдает EndpointDetails
хранит services_
ищет сервис по URI через FindServiceByUri()
Как думать о нем

Это не “устройство”. Это реестр сервисов, published services catalog.

ServiceProvider
Что это

Базовый класс для конкретного сервиса устройства.

Задача
иметь собственный endpoint типа
ws://host:port/xfs4iot/v1.0/SimCashAcceptor/
иметь собственный MessageDecoder
иметь собственный CommandDispatcher
иметь доступ к Device
уметь отправлять unsolicited events всем активным соединениям
Как думать о нем

Если ServicePublisher — это “диспетчерская”, то ServiceProvider — это “конкретное устройство/служба”.

CashAcceptorServiceProvider
Что это

Твой конкретный сервис-провайдер для cash acceptor устройства.

Задача

Собрать в одном объекте несколько интерфейсов:

Common
CashAcceptor
CashManagement
Storage
Что делает
наследуется от ServiceProvider
в Initialize() создает дочерние service-классы:
CommonServiceClass
StorageServiceClass
CashManagementServiceClass
CashAcceptorServiceClass
регистрирует команды/handlers в decoder/dispatcher
заполняет MessagesSupported
делегирует вызовы в service-классы
Как думать о нем

Это сервис-обертка над устройством, которая экспонирует XFS4IoT-команды.

EndPoint
Что это

Низкоуровневый сетевой listener.

Задача
слушать TCP порт
принимать HTTP/WebSocket upgrade
определить target path
решить, это publisher endpoint или service endpoint
создать ClientConnection
Что важно

Именно здесь у тебя происходит routing:

/xfs4iot/v1.0/ → publisher
/xfs4iot/v1.0/SimCashAcceptor/ → service provider
Как думать о нем

Это “веб-серверный вход”, который ничего не знает про XFS-бизнес-логику, кроме маршрутизации endpoint’ов.

ClientConnection (серверный)
Что это

Один активный WebSocket-сеанс с клиентом.

Задача
читать входящие WebSocket-сообщения
собирать полный message frame
передавать JSON в MessageDecoder
вызывать CommandDispatcher.Dispatch(...)
сериализовать и отправлять ответы
отменять команды при disconnect
Что делает

По сути это runtime для одного соединения:

receive loop
decode
validate
dispatch
send response
Как думать о нем

EndPoint принимает подключения, а ClientConnection уже живет внутри одного подключения.

XFS4IoTClient.ClientConnection (клиентский)
Что это

Клиентский WebSocket wrapper из sample app.

Задача
подключаться к endpoint
посылать command JSON
ждать ack/completion/event
декодировать response через MessageDecoder
Как думать о нем

Это клиентский зеркальный аналог серверного ClientConnection.

CommandDispatcher
Что это

Сердце маршрутизации команд.

Задача

По пришедшему объекту команды:

отправить Acknowledge
найти нужный handler
проверить, поддерживается ли message name/version
либо запустить handler сразу
либо поставить в очередь через CommandQueue
обработать timeout / cancel / errors
Главная мысль

Это не transport layer и не business layer. Это оркестратор выполнения команды.

Что именно делает у тебя
получает MessageBase
делает Acknowledge
через CreateHandler(typeid(*command)) ищет нужный handler
сверяет m_messagesSupported
вызывает Handle(...) или HandleError(...)
Как думать о нем

Это как контроллер маршрутов в веб-фреймворке:

path уже определен
JSON уже разобран
осталось вызвать правильный handler
CommandQueue
Что это

Очередь команд.

Задача

Нужна для команд, которые нельзя или не надо исполнять сразу параллельно.

Для чего она

Некоторые XFS-команды:

должны идти строго по очереди
не должны пересекаться
могут быть long-running
могут быть отменены
Как думать о ней

Это последовательный исполнитель/планировщик команд устройства.

MessageDecoder
Что это

Преобразователь JSON → typed message object.

Задача
прочитать header.name
выбрать зарегистрированный message type
вызвать соответствующий FromJson(...)
вернуть shared_ptr<MessageBase>
В C# и C++ разница

В C# оригинальный MessageDecoder часто автозаполняется reflection’ом по атрибутам Command, Completion, Event.
В твоем C++ варианте все регистрируется вручную: RegisterMessageType<...>(). Это важное архитектурное отличие от KAL C# framework, где reflection используется очень широко.

Как думать о нем

Это “factory по имени сообщения”.

MessageBase
Что это

Абстрактная база всех XFS4IoT сообщений.

Содержит
Header
сериализацию в JSON
регистрацию message type/name/version
общий контракт для Command / Completion / Event / Acknowledge
Как думать о нем

Это общий “конверт сообщения”.

Message<TPayload>
Что это

Шаблонная реализация сообщения с payload.

Задача

Соединить:

MessageHeader
конкретный payload type
Как думать о нем

MessageBase — это каркас, а Message<T> — уже готовое сообщение с типизированным payload.

MessageHeader
Что это

Заголовок XFS4IoT-сообщения.

Содержит
name
requestId
type
version
timeout
status
completionCode
errorDescription
Как думать о нем

Это метаданные сообщения, по которым router/decoder понимают, что делать.

Acknowledge
Что это

Промежуточный ответ сервера на принятую команду.

Зачем нужен

Когда команда принята к обработке, сервер сначала шлет acknowledge, а потом уже completion.

Как думать о нем

Это не результат команды, а подтверждение:

“команду получил, начал обрабатывать”

GetServiceHandler
Что это

Handler для ServicePublisher.GetServices.

Задача

Собрать список опубликованных service URI и вернуть GetServicesCompletion.

Как думать о нем

Это обработчик discovery-команды publisher’а.

CapabilitiesHandler
Что это

Handler для Common.Capabilities.

Задача

Собрать capabilities нескольких интерфейсов:

Common
CashManagement
CashAcceptor

и отдать CapabilitiesCompletion.

Как думать о нем

Это “витрина возможностей устройства”.

CommonServiceClass
Что это

Сервисный класс, который обслуживает XFS4IoT Common interface.

Обычно отвечает за
common capabilities
common status
common unsolicited events
error/status changed events
Как думать о нем

Это слой бизнес-логики для Common interface, а не сетевой слой.

CashManagementServiceClass
Что это

Сервисный класс для CashManagement interface.

Обычно отвечает за
cash management capabilities/status
cash-in status
present status
item classification
persistence связанного состояния
Как думать о нем

Это бизнес-логика управления денежными модулями/статусами.

CashAcceptorServiceClass
Что это

Сервисный класс именно для CashAcceptor interface.

Обычно отвечает за
capabilities acceptor-а
device lock status
replenish/deplete metadata
cash-in related device behavior
Как думать о нем

Это логика самого приемника наличных.

StorageServiceClass
Что это

Сервисный класс для Storage interface.

Задача
хранить сведения по cash units / storage units
обновлять accounting/counts
поднимать storage changed events
работать с persistence
Как думать о нем

Это inventory/storage-слой устройства.

DeviceParameters
Что это

Судя по типичной структуре KAL sample/simulator-проектов, это объект конфигурации/параметров конкретного устройства или sample device runtime. В таких проектах он обычно хранит то, что влияет на поведение симулятора: модели, лимиты, конфигурацию модулей, capabilities flags и т.п. Это согласуется с тем, что sample repo содержит device simulators и templates для SP/device code.

Как думать о нем

Это не transport и не dispatcher, а конфигурационный/модельный объект устройства.

CommonDevice (клиентская сторона)
Что это

Класс UI-уровня из sample client, который умеет:

делать service discovery
открывать connection к сервису
посылать Status, Capabilities
логировать sent/received messages в GUI
Задача

Спрятать низкоуровневый клиентский WebSocket от UI-кнопок.

Как думать о нем

Это не “устройство” сервера, а клиентский helper/view-model вокруг service endpoint.

4. Как проходит один запрос у тебя

Возьмем Common.Capabilities.

Шаг 1

Клиент уже знает service URI после GetServices.

Шаг 2

Подключается к:
/xfs4iot/v1.0/SimCashAcceptor/

Шаг 3

EndPoint принимает upgrade и routing’ом выбирает CashAcceptorServiceProvider.

Шаг 4

Создается ClientConnection.

Шаг 5

ClientConnection читает JSON:

{"header":{"name":"Common.Capabilities", ...}}
Шаг 6

MessageDecoder превращает JSON в CapabilitiesCommand.

Шаг 7

CommandDispatcher:

отправляет Acknowledge
делает CreateHandler(typeid(CapabilitiesCommand))
получает CapabilitiesHandler
Шаг 8

CapabilitiesHandler дергает:

CommonServiceClass
CashManagementServiceClass
CashAcceptorServiceClass
Шаг 9

Формируется CapabilitiesCompletion.

Шаг 10

ClientConnection::SendMessageAsync() отправляет completion клиенту.

5. Как понимать различие между классами “ServiceProvider”, “ServiceClass”, “Device”

Это часто путает.

ServiceProvider

Внешняя серверная оболочка XFS4IoT endpoint’а.

*ServiceClass

Внутренняя бизнес-логика конкретного интерфейса.

Device

Низкоуровневый симулятор/железо/adapter.

Очень грубо:

Device знает реальное состояние устройства
ServiceClass знает, как превратить состояние устройства в XFS4IoT-модель
ServiceProvider знает, как выдать это наружу по сети

Запомни так:

Publisher

“Какие сервисы у тебя есть?”

“Вот endpoint конкретного сервиса.”

EndPoint

“Принял сокет и выбрал, кому его отдать.”

ClientConnection

“Жизнь одного WebSocket-сеанса.”

MessageDecoder

“Из JSON сделал typed command.”

CommandDispatcher

“Нашел нужный handler и запустил.”

CommandQueue

“Если надо — выполняю команды последовательно.”

“Реакция на конкретную команду.”

ServiceClass

“Бизнес-логика интерфейса.”

Device

“Реальное устройство/симулятор.”

CommonDevice

“Клиентский helper для UI.”

9. Важная оговорка по “все-все”

Полную документацию вообще по каждому классу из всего KAL фреймворка лучше делать поэтапно, потому что там много проектов и интерфейсов. Но по твоей текущей задаче самое полезное — именно эта карта классов вокруг:

discovery
endpoint
connection
decoding
dispatch
handlers
service classes
provider/device separation

Именно она сейчас определяет, почему клиент подключается, делает GetServices, потом Capabilities, потом Status.



Я объясню сначала по отдельности:

Command
Completion
Event
Handler

а потом покажу несколько живых сценариев выполнения запросов.

1. Command
Что это

Command — это запрос от клиента к серверу.

То есть клиент говорит серверу:

дай список сервисов
покажи capabilities
покажи status
начни cash-in
верни storage

Все это — Command.

Примеры команд

Из твоего проекта:

ServicePublisher.GetServices
Common.Capabilities
Common.Status
Storage.GetStorage
Как выглядит логически

Команда всегда инициируется клиентом.

Например:

{
  "header": {
    "name": "Common.Capabilities",
    "requestId": 3,
    "type": "command",
    "version": "2.0",
    "timeout": 60000
  }
}
Что обычно есть у команды

У команды есть:

name — имя команды
requestId — ID запроса
type = command
version
timeout
payload — если нужны параметры
Как думать о Command

Это просто:

“Клиент попросил сервер что-то сделать или вернуть”.

2. Completion
Что это

Completion — это финальный ответ сервера на команду.

Если Command — вопрос, то Completion — окончательный ответ.

Примеры

На команды выше ответы будут:

ServicePublisher.GetServices → GetServicesCompletion
Common.Capabilities → CapabilitiesCompletion
Common.Status → StatusCompletion
Как выглядит логически

Например:

{
  "header": {
    "name": "Common.Capabilities",
    "requestId": 3,
    "type": "completion",
    "version": "2.0",
    "completionCode": "success"
  },
  "payload": {
    ...
  }
}
Что важно

Completion всегда привязан к конкретному requestId.

То есть клиент отправил:

requestId = 3

и completion должен прийти тоже с:

requestId = 3

чтобы клиент понял:

это ответ именно на тот запрос

Completion не равно Acknowledge

Это важно.

Есть два разных ответа сервера:

Acknowledge

Промежуточное подтверждение:

“команду получил, начал обрабатывать”

Completion

Финальный результат:

“обработка закончена, вот итог”

3. Event
Что это

Event — это асинхронное сообщение от сервера клиенту, которое не обязательно является прямым ответом на текущую команду.

То есть это не “ответ на вопрос”, а “уведомление о событии”.

Примеры событий

Из XFS4IoT логики:

StatusChangedEvent
StorageChangedEvent
ItemsTakenEvent
ItemsInsertedEvent
ErrorEvent
Когда нужны Event

Когда на устройстве что-то произошло само по себе:

клиент забрал деньги
кто-то вставил банкноты
изменился статус устройства
изменилось storage состояние
произошла ошибка
Как думать о Event

Это:

“Сервер сам сообщает клиенту, что что-то произошло”

а не ответ на команду.

Разница между Completion и Event
Completion

Привязан к конкретной команде.

Event

Может прийти сам по себе, независимо от команды.

4. Handler
Что это

Handler — это объект, который обрабатывает конкретную команду.

Если пришла команда:

Common.Capabilities

то ее должен обработать:

CapabilitiesHandler

Если пришла:

ServicePublisher.GetServices

то ее обрабатывает:

GetServiceHandler
Что делает handler

Handler обычно делает 2 вещи:

1. Handle(...)

Обрабатывает успешный сценарий команды

2. HandleError(...)

Формирует правильный completion при ошибке

Как думать о handler

Это аналог “controller action” в вебе.

Например:

маршрут найден
JSON уже распарсен
dispatcher уже понял тип команды
теперь handler выполняет бизнес-логику
5. Связь между ними

Теперь самое важное:

Цепочка такая
Клиент отправляет Command

↓

Сервер находит Handler

↓

Handler выполняет логику

↓

Сервер отправляет Completion

↓

В процессе или независимо могут приходить Event
6. Пример №1 — ServicePublisher.GetServices

Это самый простой и хороший пример.

Шаг 1. Клиент отправляет Command

Клиент подключается к publisher endpoint:

ws://127.0.0.1:80/xfs4iot/v1.0/

и шлет:

{
  "header": {
    "name": "ServicePublisher.GetServices",
    "requestId": 2,
    "type": "command",
    "version": "2.0",
    "timeout": 60000
  }
}

Это Command.

Шаг 2. Сервер принимает команду

Серверный ClientConnection читает WebSocket-сообщение.

Потом MessageDecoder смотрит:

header.name = ServicePublisher.GetServices

и создает объект:

GetServicesCommand
Шаг 3. CommandDispatcher ищет handler

Dispatcher делает:

CreateHandler(typeid(GetServicesCommand))

и получает:

GetServiceHandler
Шаг 4. Сервер отправляет Acknowledge

Сначала клиенту идет:

{
  "header": {
    "name": "ServicePublisher.GetServices",
    "requestId": 2,
    "type": "acknowledge",
    "version": "2.0"
  }
}
Шаг 5. Handler выполняет логику

GetServiceHandler::Handle(...):

берет список services из ServicePublisher
собирает serviceURI
создает payload
Шаг 6. Сервер отправляет Completion
{
  "header": {
    "name": "ServicePublisher.GetServices",
    "requestId": 2,
    "type": "completion",
    "version": "2.0",
    "completionCode": "success"
  },
  "payload": {
    "vendorName": "KAL",
    "services": [
      {
        "serviceURI": "ws://127.0.0.1:80/xfs4iot/v1.0/SimCashAcceptor/"
      }
    ]
  }
}
Что здесь участвовало
Command → GetServicesCommand
Handler → GetServiceHandler
Completion → GetServicesCompletion

Event тут не нужен.

7. Пример №2 — Common.Capabilities
Шаг 1. Клиент уже знает service URI

После GetServices клиент подключается к:

ws://127.0.0.1:80/xfs4iot/v1.0/SimCashAcceptor/
Шаг 2. Клиент шлет Command
{
  "header": {
    "name": "Common.Capabilities",
    "requestId": 3,
    "type": "command",
    "version": "2.0",
    "timeout": 60000
  }
}
Шаг 3. Decoder создает объект команды

MessageDecoder делает:

CapabilitiesCommand
Шаг 4. Dispatcher ищет handler

CommandDispatcher вызывает:

CreateHandler(typeid(CapabilitiesCommand))

и получает:

CapabilitiesHandler
Шаг 5. Сервер отправляет acknowledge
{
  "header": {
    "name": "Common.Capabilities",
    "requestId": 3,
    "type": "acknowledge",
    "version": "2.0"
  }
}
Шаг 6. CapabilitiesHandler собирает данные

Он идет в:

CommonServiceClass
CashManagementServiceClass
CashAcceptorServiceClass

берет оттуда capabilities и строит payload.

Шаг 7. Отправляется Completion
{
  "header": {
    "name": "Common.Capabilities",
    "requestId": 3,

8. Пример №3 — команда + события

Теперь пример, где участвует Event.

Допустим в будущем у тебя будет команда типа cash-in start.

Шаг 1. Клиент шлет Command

Например:

CashAcceptor.CashInStart
Шаг 2. Сервер отвечает Acknowledge

“Команду получил”

Шаг 3. Устройство начинает работу

Во время выполнения могут произойти события:

пользователь вставил банкноту
банкнота принята
изменился storage
изменился status

Тогда сервер может отправлять:

ItemsInsertedEvent
StorageChangedEvent
StatusChangedEvent
Шаг 4. В конце сервер шлет Completion

То есть общая схема будет:

Command
 -> Acknowledge
 -> Event
 -> Event
 -> Event
 -> Completion
Почему это удобно

Потому что long-running операция может занять время, и клиент видит промежуточные изменения состояния.

10. Самая короткая аналогия

Если совсем просто:

Command

Это “сделай / дай”

Handler

Это “кто именно это будет выполнять”

Completion

Это “готово, вот результат”

Event

Это “по дороге произошло вот это”

11. Пример на человеческом языке

Представь банкомат как человека.

Command

“Покажи свои возможности”

Handler

Сотрудник, который умеет отвечать на этот вопрос

Completion

“Я умею принимать купюры, показывать статус, работать с storage”

Event

“Кстати, пользователь только что вставил деньги”
или
“Состояние устройства изменилось”

12. Как это ложится на твой проект

В твоем сервере уже сейчас живой поток такой:

Publisher flow
GetServicesCommand
GetServiceHandler
GetServicesCompletion
Service flow
CapabilitiesCommand
CapabilitiesHandler
CapabilitiesCompletion

А потом клиент уже пытается идти дальше:

Common.Status
Storage.GetStorage

То есть архитектура уже работает именно так, как должна.

13. Главное, что надо запомнить
Команда всегда приходит от клиента

Command

Handler всегда исполняется на сервере

Handler

Completion — это финальный ответ сервера

Completion

Event — асинхронное уведомление сервера

Event
